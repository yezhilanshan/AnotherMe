# 对话信息存储方案（结合 AnotherMe-V3 现状优化版）

你原始方案方向是对的，但当前项目已经有明确的工程现实：

1. 网关后端已经采用 PostgreSQL + Redis + MinIO（并有 jobs/job_events/job_artifacts 任务模型）。
2. 前端消息页目前主要是页面内状态，调用 /api/web-search，尚未落库会话消息。
3. 课堂数据当前在 AnotherMe 侧是 JSON 文件持久化（data/classrooms），不是数据库表。

所以最优做法不是“推翻重建”，而是“在现有网关体系上扩展对话域模型”，并分阶段落地。

---

## 一、优化目标（结合现状）

1. 不破坏现有课程生成和题目视频链路。
2. 先把 AI 对话和消息中心从前端内存态变成可持久化数据。
3. 复用现有 PostgreSQL/Redis/MinIO，不引入多余中间件。
4. 保留教学场景核心价值：对话原文与学习结构化记录分层存储。

---

## 二、分层存储（项目定制版）

### 1. PostgreSQL（主存）

存放长期结构化数据：

1. 现有任务域：jobs、job_events、job_artifacts（已存在）。
2. 新增消息域：users、conversations、conversation_members、messages。
3. 新增 AI 对话域：ai_chat_sessions、ai_chat_messages、ai_message_feedback、ai_learning_records。
4. 新增行为域：event_logs。

### 2. Redis（高频缓存/实时态）

存放短期高频数据：

1. 在线状态（user:online:*）。
2. 未读数缓存（conv:unread:*）。
3. SSE/流式输出的短暂片段缓存（可选）。
4. 热门会话列表与最近会话缓存。

### 3. MinIO/S3（对象存储）

存放二进制对象：

1. 聊天附件（图片/文件/语音）。
2. 题目原图、讲题视频、调试包（你已在用）。
3. 后续导出报告 PDF。

数据库仅存 object_key、url、mime、size、checksum 等元数据。

### 4. 向量库（后续阶段）

仅在需要长期个性化记忆/RAG 时再加，不作为第一阶段阻塞项。

---

## 三、与现有系统的直接映射

### 1. 现有可复用能力

1. 任务状态追踪框架：可继续用于长耗时学习记录抽取任务。
2. 统一对象存储封装：消息附件直接复用上传逻辑。
3. 网关鉴权与 user_id 透传机制：可作为消息与对话的归属基础。

### 2. 当前空缺点（需要补）

1. /messages 页会话消息尚未落库。
2. /api/chat 是无状态 SSE，对话上下文未沉淀到数据库。
3. 课堂数据与对话数据目前是两套存储（文件 vs 数据库），缺少关联字段。

### 3. 优化原则

1. 先补会话与消息落库，再谈复杂推荐。
2. 课堂存储暂不强制迁移数据库，但新增可选 classroom_id 外键关联。
3. 学习分析通过异步抽取，不阻塞主链路回复时延。

## 四、推荐表结构（在你原方案基础上收敛）

### 1. 消息中心核心表

1. users
2. conversations
3. conversation_members
4. messages
5. message_attachments

关键补充字段建议：

1. messages.source_type：manual / ai / system / classroom。
2. messages.source_ref_id：可关联 ai_message_id 或 classroom_id。
3. conversation_members.last_read_seq：建议用递增序号做已读游标，比按 message_id 更稳。

### 2. AI 对话核心表

1. ai_chat_sessions
2. ai_chat_messages
3. ai_message_feedback
4. ai_learning_records

关键补充字段建议：

1. ai_chat_sessions.linked_classroom_id（可空）：打通课堂与对话。
2. ai_chat_messages.request_id：防止前端重试导致重复入库。
3. ai_chat_messages.parent_message_id：支持追问链。
4. ai_learning_records.extract_version：记录抽取器版本，便于回灌重算。

### 3. 行为与审计

1. event_logs：业务行为埋点（打开会话、点赞、收藏、分享）。
2. 继续使用 job_events：保留任务执行侧审计（生成链路进度）。

这样可避免把“行为分析日志”和“任务执行日志”混在一起。

## 五、索引与约束（必须落地）

建议至少加以下约束：

1. conversations(type, updated_at desc) 复合索引。
2. conversation_members(conversation_id, user_id) 唯一约束。
3. messages(conversation_id, seq) 唯一约束，seq 递增。
4. ai_chat_messages(session_id, created_at) 索引。
5. ai_chat_messages(request_id) 唯一约束（幂等）。
6. ai_learning_records(user_id, subject, knowledge_point, created_at) 复合索引。

3. GET /api/messages/{conversationId}/messages

4. POST /api/ai/sessions/{sessionId}/messages

### 3. 学习记录异步化

新增 job_type：learning_record_extract（建议），由 worker 异步抽取 ai_learning_records。

## 七、四阶段落地计划（最符合当前仓库）

### Phase 1（1-2 周）：AI 对话先落库

1. 新建 ai_chat_sessions、ai_chat_messages、ai_message_feedback。
2. 保持现有 /api/chat SSE 机制，在服务端补充写库。
3. 前端消息页优先读取数据库会话，不再仅依赖页面内状态。

### Phase 2（2-3 周）：消息中心正式化

1. 新建 conversations、conversation_members、messages、message_attachments。
2. 接入未读数与已读游标（Redis 缓存 + PostgreSQL 落盘）。
3. 打通“分享 AI 回答到消息会话”的 source_type=ai_share。

### Phase 3（2 周）：学习数据抽取

1. 新建 ai_learning_records、event_logs。
2. 对已完成 ai_chat_messages 做异步抽取入库。
3. 实现基础报表：学科分布、高频知识点、困惑点趋势。

### Phase 4（可选）：长期记忆与推荐

1. 新建 student_profiles、student_memory。
2. 按需接入向量库，做个性化检索增强。

## 八、与课堂数据的关系（当前必须说明）

当前课堂是文件存储，不建议立刻迁移数据库。建议采用“软关联”：

1. ai_chat_sessions.linked_classroom_id 可关联 classroom id。
2. messages.source_ref_id 可记录 classroom id 或 scene id。
3. 等课堂存储后续升级时，再做强外键迁移。

这样既能追踪“这段对话服务于哪堂课”，又不会引入大规模迁移风险。

---

## 九、风险点与规避策略

1. 重试导致重复消息：用 request_id 唯一约束 + 幂等写入。
2. 高并发未读数错乱：Redis 做增量，定时与数据库对账。
3. 大文本消息膨胀：正文表 + 可选归档表分层。
4. 附件泄露风险：对象存储使用短时签名 URL，不暴露永久公网链接。
5. 学习抽取误差：记录 extract_version，支持离线重跑回填。

## 十一、一句话结论（针对你当前项目）

最适合 AnotherMe-V3 的方案是：在现有 PostgreSQL + Redis + MinIO + Job 架构上增量扩展消息中心和 AI 对话数据模型，先完成会话消息持久化，再异步沉淀学习记录，最后按需引入向量记忆，从而在不破坏现有课程与题目视频链路的前提下实现教学数据闭环。
