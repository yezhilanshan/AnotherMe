这 3 个问题可以统一归结为一句话：

**当前系统里，音频脚本、公式脚本、几何脚本，没有共享同一个“步骤级真值状态”。**

所以才会出现：

* 音频说了，公式没出
* 图里多了不该有的线
* 动画自己乱动，和讲解脱节

我建议你不要继续头痛医头，而是直接把视频生成链路改成 **“单一步骤脚本驱动三端输出”**。

---

# 一、先逐条判断本质问题

## 1）公式展示太少，只展示最终结果

这不是简单的“少渲染几个公式”，而是：

### 根因

当前公式更像是“结果摘要”，不是“推导过程脚本”。

也就是说，音频在按“过程”讲：

* 已知什么
* 推出什么
* 再代入什么
* 最终得到什么

但公式展示模块拿到的可能只是：

* 最终表达式
* 或少量关键式

于是画面就只剩“结论型公式”，缺少中间过渡。

### 本质失配

**音频是 step-by-step，公式是 result-only。**

---

## 2）`coordinate_scene.ggb.txt` 里出现原题没有的线段

这说明当前几何场景文件不是“受约束生成”，而是“自由扩展生成”。

### 根因

很可能有这几种情况：

#### 情况 A：模型为了“补全图形”自动加线

它认为：

* 菱形就该把所有边都补全
* 折叠就该画某些连线
* 角关系就自动生成辅助结构

但这些线：

* 不是原题已知
* 也不是当前步骤需要的辅助线

#### 情况 B：几何中间表示缺乏“线段来源标签”

现在系统里线段可能只是：

* 被识别到的线
* 被推理出的线
* 被模型顺手补的线

但没有区分：

* `given` 原题已有
* `auxiliary` 合法辅助线
* `derived` 推导中临时关系
* `invalid` 不该画出来的补线

所以最终导出时把不该显示的也混进去了。

#### 情况 C：场景更新是“追加式”，不是“白名单式”

也就是：

* 旧图上不断增量添加元素
* 没有严格检查“当前时刻允许显示什么”

于是历史脏元素残留到了最终视频里。

---

## 3）几何动画和音频不一致，经常乱动/该动不动

这个是最核心的问题。

### 根因

现在动画生成大概率是“看到语义后自主发挥”，而不是“严格执行动作指令”。

也就是说：

* 音频脚本说“连接 DE”
* 动画模块理解成“那我顺便高亮几个角，再移动一下点，再闪一下边”
* 或音频说“观察全等”
* 动画模块没有拿到明确动作，就什么都不做

### 本质失配

**音频是语义级，动画是自由生成级。中间缺少动作编排层。**

所以你看到的就是：

* 莫名其妙的动画
* 关键步骤没有动画
* 节奏和时机不对

---

# 二、最核心的改法：引入“步骤级导演脚本”

你现在最该补的不是再调 prompt，而是加一层：

## `step_script` / `director_script`

每一步都必须显式定义三件事：

1. **这一步说什么**（音频）
2. **这一步屏幕上显示什么公式**（公式）
3. **这一步几何图做什么动作**（动画）

也就是把“讲解”从自由生成，改成**结构化分镜**。

---

## 你可以定义成这种结构

```json
{
  "step_id": 3,
  "narration": "因为ABCD是菱形，所以AB等于BC，且AB平行CD。",
  "formula_blocks": [
    {"id": "f1", "latex": "AB = BC"},
    {"id": "f2", "latex": "AB \\parallel CD"}
  ],
  "geometry_actions": [
    {"type": "highlight_segment", "target": "AB"},
    {"type": "highlight_segment", "target": "BC"},
    {"type": "highlight_parallel", "target": ["AB", "CD"]}
  ],
  "visible_objects": {
    "points": ["A", "B", "C", "D", "E"],
    "segments": ["AB", "BC", "CD", "DA", "DE"],
    "angles": [],
    "polygons": ["ABCD"]
  }
}
```

这样一来：

* 音频不再单独生成
* 公式不再单独猜
* 动画不再自由发挥

而是三者都从同一个 step script 里读。

---

# 三、针对 3 个问题的直接修复方案

---

## 问题 1：公式展示不全

### 直接改法

把公式输出从“最终结果列表”改成“步骤级公式块列表”。

### 不要再只存：

```json
{
  "final_formula": "x = 5"
}
```

### 而要存：

```json
{
  "step_id": 1,
  "formula_blocks": [
    {"latex": "AD = 5"},
    {"latex": "\\tan B = 2"}
  ]
},
{
  "step_id": 2,
  "formula_blocks": [
    {"latex": "\\tan B = \\frac{AD}{AB} = 2"},
    {"latex": "AB = \\frac{5}{2}"}
  ]
},
{
  "step_id": 3,
  "formula_blocks": [
    {"latex": "BE \\perp EB'"},
    {"latex": "\\angle BEB' = 90^\\circ"}
  ]
}
```

---

### 展示策略也要改

不是“一步只显示一个最终式”，而是支持：

* 新公式 `write`
* 旧公式保留 `keep`
* 变形式 `transform`
* 不再需要的公式 `fade_out`

例如：

```json
{
  "formula_actions": [
    {"type": "write", "latex": "\\tan B = \\frac{AD}{AB}"},
    {"type": "transform", "from": "\\tan B = \\frac{AD}{AB}", "to": "2 = \\frac{5}{AB}"},
    {"type": "transform", "from": "2 = \\frac{5}{AB}", "to": "AB = \\frac{5}{2}"}
  ]
}
```

### 这样能解决什么

* 音频讲到哪，公式就展示到哪
* 不会只剩最后答案
* 用户能看到“推导链”

---

## 问题 2：图里出现不必要线段

这个要做**几何对象白名单 + 来源管理**。

---

### 第一层：给每个几何对象打来源标签

例如：

```json
{
  "id": "seg_DE",
  "type": "segment",
  "from": "D",
  "to": "E",
  "source": "given"
}
```

或：

```json
{
  "id": "seg_AC",
  "type": "segment",
  "from": "A",
  "to": "C",
  "source": "auxiliary",
  "reason": "used_for_triangle_congruence"
}
```

禁止无标签对象进入最终场景。

---

### 第二层：导出 `coordinate_scene.ggb.txt` 前做过滤

只允许两类线进入最终图：

* 原题已知线段 `given`
* 当前步骤明确批准的辅助线 `approved_auxiliary`

例如：

```python
ALLOWED_SOURCES = {"given", "approved_auxiliary"}

scene.segments = [
    s for s in scene.segments
    if s.source in ALLOWED_SOURCES
]
```

---

### 第三层：辅助线必须“按需申请”

不要让模型自由加线。

应该变成：

```json
{
  "auxiliary_request": {
    "segment": "AC",
    "reason": "need diagonal to prove triangle congruence",
    "step_id": 4
  }
}
```

然后由规则层审核：

* 是否真的必要
* 是否与当前题型匹配
* 是否在当前步骤讲到了

通过后才加入场景。

---

### 第四层：场景快照改成“重建式”，不要“累计式”

不要每步都在上一步基础上乱加对象。

应该每一步都根据当前 `visible_objects` 重建画面：

```python
current_scene = build_scene_from_visible_objects(step.visible_objects)
```

而不是：

```python
scene.add(...)
scene.add(...)
scene.add(...)
```

这样能彻底消除“前面误加的脏线残留”。

---

## 问题 3：动画和音频不一致

这个必须加 **动作语法层**，不要让动画模块自己猜。

---

### 把动画限制为有限动作集合

例如只允许这些原子动作：

* `show_point`
* `show_segment`
* `show_auxiliary_segment`
* `highlight_segment`
* `highlight_angle`
* `highlight_triangle`
* `move_point`
* `reflect_shape`
* `show_formula_relation`
* `fade_out_object`

这样动画模块就不是“生成动画”，而是“执行动作序列”。

---

### 每句音频绑定动作

例如：

```json
{
  "narration": "连接DE作为折叠轴。",
  "geometry_actions": [
    {"type": "show_segment", "target": "DE"},
    {"type": "highlight_segment", "target": "DE"}
  ]
}
```

```json
{
  "narration": "将点B折叠到B′的位置。",
  "geometry_actions": [
    {"type": "reflect_point_over_line", "target": "B", "axis": "DE", "output": "B1"},
    {"type": "show_point", "target": "B1"},
    {"type": "show_segment", "target": "EB1"}
  ]
}
```

---

### 动画必须有触发条件

不是每步都自动动。

建议加字段：

```json
{
  "animation_policy": "required"
}
```

或：

```json
{
  "animation_policy": "none"
}
```

这样：

* 只是口头说明时，不乱动
* 真到构造、翻折、连线、标角时，必须动

---

### 给动作加时间锚点

例如：

```json
{
  "audio_span": [12.5, 15.2],
  "geometry_actions": [
    {"type": "show_segment", "target": "DE", "at": 12.8},
    {"type": "highlight_segment", "target": "DE", "at": 13.1}
  ],
  "formula_actions": [
    {"type": "write", "latex": "DE\\text{ 为折叠轴}", "at": 13.0}
  ]
}
```

这样才能真正做到同步，而不是“同一步大概一致”。

# 四、你现在最值得优先改的模块顺序

按收益排序，我建议你先做这 4 个。

## 第一优先级：统一步骤脚本

新增一个中间产物：

* `lesson_plan.json` 或 `director_script.json`

它是整个视频生成唯一真源。

没有这个，后面都只是补丁。

---

## 第二优先级：几何对象白名单机制

重点修 `coordinate_scene.ggb.txt` 污染问题。

你要做到：

* 所有线段都可追踪来源
* 所有辅助线都必须审批
* 每一步导出前都过滤

这一步做完，图里乱线问题会立刻下降很多。

---

## 第三优先级：公式从“结果列表”改成“过程块”

这一步做完，公式不足的问题会明显改善。

关键不是多显示，而是：

* 按步骤显示
* 按推导变形显示
* 与音频共享同一 step

---

## 第四优先级：动画从“生成式”改成“执行式”

把“动画 agent”降权，让它不再自由创作，只负责：

* 把结构化动作翻译成 Manim / 渲染指令

这是解决“乱动”的关键。

# 六、给你一个最小可落地版本

如果你现在不想大改全链路，可以先做一个 **MVP 修复版**：

## MVP-1：给现有每个步骤补 `visible_segments`

```json
{
  "step_id": 2,
  "visible_segments": ["AB", "BC", "CD", "DA", "DE"]
}
```

导出 `coordinate_scene.ggb.txt` 时只保留这些线段。
这能立刻减少多余线段。

---

## MVP-2：给每步补 `spoken_formulas`

```json
{
  "step_id": 3,
  "spoken_formulas": [
    "AB = BC",
    "\\angle BEB' = 90^\\circ"
  ]
}
```

公式展示直接从这里读。
这样能先保证“音频说过的公式至少能上屏”。

---

## MVP-3：给每步补 `required_actions`

```json
{
  "step_id": 4,
  "required_actions": [
    {"type": "show_segment", "target": "DE"},
    {"type": "highlight_angle", "target": "BEB1"}
  ]
}
```

先只允许执行 `required_actions`，禁掉自由补动画。
这样能快速压住“乱动”。

# 七、最后给你一个结论

你现在的问题不是某个模型“不聪明”，而是**生产链路缺少导演层**。

要解决，核心不是继续让模型自由生成更多内容，而是：

**让模型先生成“步骤级导演脚本”，再由公式模块、几何模块、音频模块共同执行。**

这样三件事会一起变好：

* 公式会更完整
* 图里不会乱长线
* 动画会和音频真正对齐
