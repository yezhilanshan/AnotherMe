这份逻辑的核心问题很明确：

**入口是保守的，但中间融合和补全阶段又把“推断事实”重新注入回主流程了。**

所以表面上你写了“Be conservative, omit instead of guessing”，但系统行为上并没有真正做到保守。代码里最关键的问题有 6 个。

---

## 1. 你现在仍然是“单个 VL 主导”，不是双路互证

你在 `_analyze_problem_bundle()` 里，主入口还是一次性让几何模型同时做：

* OCR
* geometry_facts 提取

也就是这个 prompt：
“Do OCR accurately, then extract geometry entities, relations, and measurements” 

后面虽然又调用了 `_extract_problem_text_fallback()` 做 OCR fallback，但它本质只是：

* 用 OCR 路替换 problem_text
* **没有把 OCR 的结构化条件，变成一套独立的几何事实源**

也就是说：

* OCR 路只负责“文本替换”
* VL 路负责“几何事实定义”
* 两者没有真正对齐到同一中间表示

所以这不是“两个 agent 互补”，而是“一个主 agent + 一个文本兜底”。这就是你说的“两者没有互补”的直接原因。

---

## 2. 最大的问题：你把 inferred_relations / inferred_measurements 合并回主 facts 了

这段是最关键的风险点：

在 `_compose_compiler_geometry_facts()` 里，你把：

* `inferred_relations`
* `inferred_measurements`

筛一遍后直接 merge 到：

* `merged["relations"]`
* `merged["measurements"]` 

这意味着什么？

意味着只要文本增强阶段推出来的关系，最后就和“原始观测关系”混到同一层了。
虽然你保留了 source，但对后面的 compiler 来说，它们已经都是正式输入。

这会直接导致两个后果：

### 第一种错误

题干出现“菱形ABCD”，你自动补出：

* 平行
* 四边相等

这些其实属于**定义推导事实**，不是图中观测事实。
但你现在把它们混进了 `relations`。

### 第二种错误

题干出现 `tanB=2`，你自动补：

```python
{"type": "angle", "angle": angle_name, "value": f"arctan({match.group(2)})"}
```

这也是推导值，不是题图直接给出的角度。
但你又把它放进了 measurements 主桶。

这就造成了：

**系统没有区分“看见的”“题干明确说的”“根据定义推导的”“根据解题过程可能推出的”。**

这是当前最核心的结构问题。

---

## 3. `_augment_facts_from_problem_text()` 太激进，会自动补大量高阶几何关系

你这个函数不是“轻微补全”，而是在做**规则推理**，而且推理结果会进入主 facts。

尤其是这些地方风险很高：

### 菱形自动补性质

你只要识别到“菱形ABCD”，就自动补：

* 两组平行
* 多组 equal_length 

这在“求解阶段”是可以的，但在“视觉建模阶段”不应该直接进入主 facts。

### 圆相关自动补

只要文本出现 `⊙O` 或 “P 在 ⊙O 上”，你就自动建立：

* circle_O
* point_on_circle 关系 

这个比菱形问题小一点，因为这是文本显式条件，问题不大；
但也应该标成 `text_explicit`，而不是跟视觉观测同层。

### tan 自动补角

这是更危险的地方。
`tanB=2` 不是“角B已知具体值”，只是一个三角函数关系。你直接变成 angle measurement，会误导后续编译器和动画。

所以这层不该叫 augment，更像是 **symbolic inference**。它不该写回 observed facts。

---

## 4. 你的 sanitize 是“清洗格式”，不是“约束裁决”

`_sanitize_geometry_facts()` 做了很多规范化工作：

* 点名清洗
* 线段名清洗
* angle bucket 清洗
* relation bucket 清洗
* measurement bucket 清洗 

这些都很好。

但问题是，它主要在做：

* schema 对齐
* token 合法化
* 去重

它**没有做真正的证据分级**。比如：

* 这个垂直是来自直角符号吗？
* 这个 equal_length 是刻痕看到的，还是因为“像菱形”？
* 这个 midpoint 是题目说的，还是模型猜的？

目前没有统一字段去表示：

* source
* evidence
* status
* certainty tier

所以 sanitize 完之后，所有关系都长得差不多。
这会让后面的 compiler 根本不知道哪些能信、哪些只能参考。

---

## 5. 后续 compiler / scene solver 吃的是“混合事实”，所以会越走越偏

`process()` 的主链路是：

* `_extract_and_stabilize_bundle()`
* `_compile_and_infer()`
* `normalize_geometry_spec`
* `solve_coordinate_scene` 

问题在于 `_compile_and_infer()` 之前，facts 已经被文本推导增强过，而且 compiler 还会继续吸收 inferred facts。

这样就形成了一个典型链条：

**视觉猜一点 + 文本推一点 + compiler 再补一点 + solver 再按这些去解**

于是最后的 coordinate_scene 看起来“很完整”，但里面混了不少并非原题明确给出的东西。
这就是你说的：

> 对于题目中没有给出的一些角度等信息也要自行推断出来

本质原因不是单个模型太自信，而是**系统允许推断事实直接进入几何建模主链路**。

---

## 6. 你其实已经意识到“折叠题高风险”，但处理还是偏后置

你代码里已经专门处理了：

* `_contains_fold_semantics()`
* `_fold_solver_failed()`
* `_prune_fold_reflection_artifacts()`
* `_assess_schematic_scene_policy()` 

说明你已经发现折叠、圆、动态点这些题非常容易幻觉。
但这些保护大多发生在：

* compile 之后
* solve 失败之后
* fallback 阶段

也就是说，你是在**后面补救**，不是前面拦截。

更合理的是：

* 在 facts 层就降低推断权限
* 高风险题型默认禁止 inferred 几何关系进入 compiler
* 只允许 text_explicit / diagram_marker / very_high_confidence observed 进入

这样会比后面 prune artifact 更干净。

# 直接下结论：你现在逻辑的真正 bug 在哪

一句话概括：

**你现在把“观测”“文本显式条件”“规则推导”“求解推断”四层事实混成了两层，甚至常常混成一层。**

于是系统会表现成：

* 表面保守
* 实际激进
* 越往下游越确信
* 最终生成了原题没明确给出的关系

# 我建议你怎么改，优先按最小改动来

## 第一优先级：不要把 inferred_* 合并回 relations / measurements

也就是改 `_compose_compiler_geometry_facts()`。

现在是：

* inferred_relations → merged["relations"]
* inferred_measurements → merged["measurements"] 

建议改成：

* `observed_relations` 单独保存
* `text_explicit_relations` 单独保存
* `derived_relations` 单独保存
* compiler 默认只吃前两类
* derived 仅在特定模板下按白名单启用

这是收益最大的一刀。

---

## 第二优先级：给每条事实加 provenance

建议每个 relation / measurement 都带：

```json id="96839"
{
  "type": "parallel",
  "segments": ["AB", "CD"],
  "source": "text_derived",
  "status": "derived",
  "confidence": 0.82,
  "evidence": ["problem_text: 菱形ABCD"]
}
```

最少加这三个字段：

* `source`
* `status`
* `evidence`

这样 compiler 和 animation 才能知道哪些能直接用。

---

## 第三优先级：把 `_augment_facts_from_problem_text()` 改成“旁路输出”

现在它直接写回：

* `facts["relations"]`
* `facts["measurements"]` 

建议改成：

* `facts["derived_relations"]`
* `facts["derived_measurements"]`

不要再写回主桶。

---

## 第四优先级：高风险题型禁止自动补几何关系

对这些 pattern：

* `fold_transform`
* `circle_geometry`
* `dynamic_point`

默认规则：

* 禁止 inferred_relations 进入 compiler
* 禁止 symbolic angle 进入 solver
* 禁止仅由题型名推出的附加约束进入 coordinate_scene

你已经有 `_assess_schematic_scene_policy()` 了，可以直接把这层前移。

---

## 第五优先级：让 OCR 路输出结构化文本条件，不只是 problem_text

现在 OCR 路只是文本兜底。
建议新增一层专门做：

* `text_entities`
* `text_relations`
* `text_measurements`

也就是从 OCR 文本中单独抽：

* “菱形ABCD”
* “AD=5”
* “tanB=2”
* “E是AB上一点”
* “沿DE折叠”

这样它才能和 VL 的：

* points
* markers
* segments
* relations

在融合层真正互补，而不是只替换 problem_text。
