## 第2页：项目背景（现有AI辅助教学痛点）
1. 内容生产成本高：教师备一节完整课，需要同时准备讲解材料、练习题、互动环节、课后反馈，工作量大。
2. 多数AI工具“只会答题，不会上课”：停留在单轮问答，缺少完整课堂组织能力。
3. 学习链路割裂：课前备课、课堂互动、课后答疑、学习复盘往往分散在不同系统。
4. 题目讲解不直观：图形题、几何题仅文字讲解理解门槛高，缺少可视化推导过程。
5. 长耗时AI任务不可控：生成过程黑盒、失败不可追踪，影响教学场景可用性。

- 右侧“痛点词云”（高成本、低互动、碎片化、不可追踪）

## 第3页：项目目标
1. 构建一体化AI教学平台：打通课堂生成、互动课堂、拍题答疑、学习分析闭环。
2. 从“问答工具”升级为“课堂引擎”：支持Slides、Quiz、Interactive、PBL多场景自动编排。
3. 支持多模态表达：文本、图像、语音、动画、视频协同输出，提升理解效率。
4. 实现工程可落地：统一网关、异步任务编排、状态可观测、失败可恢复。
5. 面向规模化推广：支持本地开发与Docker部署，便于校内外复制落地。

---

## 第4页：解决思路（对应痛点）
1. 针对“备课成本高”：
	方案是主题/PDF一键生成课堂，自动形成课程结构与场景内容。
2. 针对“AI缺少课堂组织能力”：
	方案是引入多智能体角色（教师/学生/用户）协同表达，增强课堂交互性。
3. 针对“学习链路割裂”：
	方案是统一任务与数据流，形成生成-互动-答疑-复盘的完整路径。
4. 针对“题目讲解不直观”：
	方案是拍题后走视觉解析到动画视频链路，实现步骤化可视讲解。
5. 针对“任务不可控”：
	方案是FastAPI网关+队列Worker+状态轮询，做到可追踪、可重试、可审计。

本页建议采用“痛点-策略-价值”三列表格。

## 第5页：方案总览（项目功能全景）
1. 课堂生成：输入主题/要求/PDF，自动生成课堂并进入回放。
2. 课堂播放：Stage + Scenes组织课堂，覆盖讲解、测验、互动、项目任务。
3. 拍题答疑：上传题目图片，自动生成讲解脚本、语音、动画和视频结果。
4. 交流讨论：支持消息会话与多角色课堂发言展示，形成讨论氛围。
5. 总结分析：统计学习时长、课堂数量、能力雷达图、薄弱项建议。
6. 统一任务平台：上传、建任务、查状态、查结果的标准化API。

可在本页加一张“功能地图”：
输入层（主题/PDF/题图）-> 生成层（课堂/视频）-> 交互层（讨论/回放）-> 评估层（统计/建议）

## 第6页：技术架构
1. 前端交互层（Next.js 16 + React 19 + TypeScript）：
	负责课堂创建、生成预览、互动课堂、拍题视频、统计分析等页面。
2. 业务编排层（API Gateway / FastAPI）：
	提供统一接口，负责参数校验、任务入队、进度更新、结果归档。
3. 执行层（Worker + 引擎）：
	分发并执行课程生成与题目视频生成任务。
4. AI引擎层（LangGraph多智能体工作流）：
	完成视觉理解、脚本生成、语音合成、动画生成、修复与合并。
5. 基础设施层：
	PostgreSQL存任务元数据，Redis做队列，MinIO/S3存制品。

建议在本页展示分层架构图与关键端口（3000/8080/5432/6379/9000）。

## 第7页：解决方法一（课堂生成链路）
1. 入口：用户填写课程主题、补充要求，可选上传PDF资料。
2. 解析：系统调用PDF解析接口提取文本与图像信息。
3. 生成：调用课堂生成任务，开启Web搜索、图像、视频、TTS等能力。
4. 编排：将内容组织为可播放场景（slide/quiz/interactive/pbl）。
5. 回放：生成完成后自动跳转课堂页面，支持沉浸式学习体验。

价值：把“备课工作流”压缩为“输入需求 -> 自动生成 -> 进入课堂”。

---

## 第8页：解决方法二（拍题答疑链路）
1. 上传题图：通过上传接口把题目图片送入网关。
2. 任务创建：生成problem_video任务并进入异步处理。
3. 引擎执行：Vision -> Script -> Voice -> Animation -> Repair -> Merge。
4. 过程可视：前端展示上传、识别、排队、语音生成、渲染等阶段进度。
5. 结果输出：返回可播放视频URL、时长、步骤数，支持回看与收藏。

价值：把抽象推导转化为“可看、可听、可复习”的讲题视频。

---

## 第9页：解决方法三（工程稳定性与可运营）
1. 统一Job协议：支持course_generate、problem_video_generate、study_package_generate。
2. 队列执行：按优先级消费任务，提升系统吞吐与资源利用率。
3. 状态管理：queued/running/succeeded/failed 全状态可追踪。
4. 结果归档：生成后登记结果与制品，便于后续查询与复用。
5. 部署模式：本地脚本一键拉起，Docker Compose统一编排，便于演示与迁移。

价值：从“能跑Demo”升级到“可持续运行的产品原型”。

## 第14页：市场分析
1. 需求侧趋势：
	AI教育正从“工具化问答”走向“场景化教学服务”，学校、机构和个人学习者都需要更完整的教学闭环。
2. 目标用户：
	中小学教师、教培机构、课后辅导场景、家庭学习用户。
3. 竞争对比（方向性）：
	通用大模型强在问答，本项目强在课堂编排+拍题视频+学习分析的一体化。
4. 差异化优势：
	多智能体课堂组织、题图到视频闭环、统一任务网关、工程化可部署。
5. 商业化路径：
	校内试点SaaS订阅、机构版私有化部署、教学资源增值服务。

## 第15页：可行性分析
1. 技术可行性：
	已有可运行前后端与网关Worker体系，关键链路已打通，不是纸面方案。
2. 产品可行性：
	覆盖“课前生成-课中互动-课后答疑-复盘分析”完整教学流程。
3. 运营可行性：
	支持本地与容器部署，便于学校试点与阶段性扩容。
4. 成本可行性：
	采用异步任务和模块化服务，可按需开启能力，控制调用与算力成本。
5. 风险与对策：
	内容准确性风险 -> 教师审核与模板约束；
	长链路稳定性风险 -> 重试机制与可观测监控；
	数据安全风险 -> 令牌鉴权与对象隔离存储。

收尾金句（可用于结语）：
AnotherMe 不是“再做一个AI问答”，而是把教学流程重构为可生成、可互动、可追踪、可优化的智能课堂基础设施。
