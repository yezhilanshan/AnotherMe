本文基于当前仓库实现核对后给出，重点不是重写已有系统，而是在现有 `Geometry IR + Teaching IR + 动画执行链路` 基础上，推动其从“可用”升级为“稳定、可扩展、可持续优化”。

## 一、当前判断：项目已进入“结构升级”阶段，而非“功能补洞”阶段

当前项目并不是从零开始，以下关键能力已经具备基础实现：

* 已有 `Geometry IR` 与 `Teaching IR` 双层结构
* 已有步骤级场景更新
* 已有折叠题相关动作抽取与映射
* 已有基础辅助线策略引擎
* 已有部分点位移动、步骤动画和模板化代码生成能力

因此，当前主问题已经不是：

* 能不能生成动画
* 能不能识别折叠
* 能不能做步骤化演示

而是：

**这些能力还没有形成“前置结构化驱动 + 可校验执行 + 可反馈迭代”的主线体系。**

这会导致系统在复杂题、变换题、长动作链路场景下仍然容易出现：

* 动作不稳定
* 推理与演示脱节
* 动画能生成但不够像老师讲题
* 失败样本无法有效沉淀和迭代

---

## 二、本次升级后的核心目标

将当前链路：

**题图识别 → 文本脚本 → 动画侧反推动作**

升级为：

**题图识别 → 题型识别层 → Geometry IR（事实） → Teaching IR（动作） → 动作可执行性校验 → 脚本/动画双消费 → 数据闭环回流**

其中新增的三个关键模块为：

1. **题型识别层（Problem Pattern Layer）**
2. **动作可执行性校验器（Action Executability Checker）**
3. **可学习的数据闭环（Case Replay & Improvement Loop）**

## 三、先回答：当前你列出的原始问题，在升级后应如何重新定义

### 1）仍然是“识别题目 → 生成结论 → 静态贴图”吗？

不是纯静态贴图，但当前仍存在“动作表达滞后于文本表达”的问题。
系统虽然已具备步骤级场景更新和局部动画能力，但动作链路还没有完全成为上游主线。

### 2）缺少“步骤级几何动作脚本”吗？

不是完全缺少，而是**已有但不够强约束**。
Teaching IR 已能承载 step-level actions，但它目前更多是动画侧推断结果，而不是脚本侧直接产出的核心契约。

### 3）折叠题等变换题没有专门动作模板吗？

已有基础模板化处理，但缺少模板族、模板匹配和模板级可验证执行机制，因此复杂折叠题仍容易不稳。

### 4）缺少辅助线策略引擎吗？

已有雏形，但还没有形成“候选生成 → 评分 → 执行 → 验证”的完整主线。

### 5）缺少 Geometry IR + Teaching IR 双层结构吗？

这一项已经不是当前主问题。
双层 IR 已落地，下一步重点应是：

* 在其上增加题型识别层
* 增加动作执行校验
* 增加数据回流闭环

## 四、当前项目最核心的瓶颈

当前最大的瓶颈不是“有没有 IR”，而是：

### 瓶颈一：缺少前置题型识别层

导致后续动作生成、模板匹配、辅助线决策没有明确“演示范式”约束。

### 瓶颈二：动作链路缺少可执行性校验

导致 Teaching IR 即使“看起来合理”，也可能在动画执行时出现依赖错误、对象缺失、时序错位。

### 瓶颈三：失败样本没有形成有效数据闭环

导致问题只能靠人工观察和局部修补，难以系统性迭代。

## 五、本项目升级后的总体架构

升级后的主线建议为：

**题图输入**
→ **视觉识别 / OCR / 文字结构抽取**
→ **题型识别层（Problem Pattern Layer）**
→ **Geometry IR（事实层）**
→ **Teaching IR（动作层）**
→ **动作可执行性校验器**
→ **脚本生成 / 动画执行 / 场景更新**
→ **结果落盘与案例回流**

三层含义如下：

### 1. Problem Pattern Layer：题型识别层

负责判断这道题属于哪种“教学演示范式”。

### 2. Geometry IR：事实层

负责表达题目中真实存在的点、线、图形、关系、变换、不变量、求解目标。

### 3. Teaching IR：动作层

负责表达讲解视频中应该如何展示、如何构图、如何突出重点、如何推进步骤。

## 六、新增模块一：题型识别层（Problem Pattern Layer）

### 1. 为什么必须新增

当前很多问题不是 Geometry IR 不够，而是系统不知道这题究竟应该按什么“演示模式”来讲。

例如以下几类题，视觉表现方式完全不同：

* 静态证明题
* 度量计算题
* 折叠变换题
* 动点轨迹题
* 相似全等题
* 圆综合题
* 立体几何截面题

如果没有题型识别层，后续模块只能“通用生成”，结果往往是：

* 动作不聚焦
* 画面节奏不对
* 辅助线加得散
* 该强调的不变量没有提前强调

### 2. 建议输出内容

```json
{
  "pattern_version": "v1",
  "problem_pattern": "fold_transform",
  "sub_pattern": "fold_with_perpendicular_construction",
  "presentation_mode": "transformation_first",
  "reasoning_mode": "invariant_then_auxiliary",
  "confidence": 0.92
}
```

### 3. 建议支持的一级题型

建议至少先覆盖：

* `static_proof`
* `metric_computation`
* `fold_transform`
* `dynamic_point`
* `similarity_congruence`
* `circle_geometry`
* `solid_geometry_section`

### 4. 题型识别层的作用

题型识别层一旦产出，可直接影响：

* Teaching IR 模板选择
* 辅助线候选生成范围
* 讲解节奏
* 动作优先级
* 动画镜头策略
* 失败时的降级方案

---

## 七、新增模块二：动作可执行性校验器（Action Executability Checker）

### 1. 为什么必须新增

当前系统即使能生成 Teaching IR，也并不等于它一定“能顺利执行”。

很多失败并不是几何逻辑错误，而是动画动作链路本身有问题，例如：

* 某对象还没创建就被引用
* 某像点还未生成就先被高亮
* 某辅助线未构造就要求做标注
* 某步骤的依赖对象只存在于前一步临时场景中
* 动作顺序与场景状态不一致

所以必须在“进入代码生成 / 场景执行”之前增加一层动作校验。

### 2. 校验范围

动作可执行性校验器建议覆盖：

#### 对象依赖合法性

* action 引用的点、线、图元是否已存在
* 新对象是否在本步中被合法创建

#### 时序合法性

* 先创建后使用
* 先变换后标注
* 先构造辅助线后做距离说明

#### 场景一致性

* 当前 step 的场景状态是否满足 action 的执行前提
* 上一步产生的对象是否在当前 step 仍然有效

#### 渲染安全性

* 标签是否过密
* 是否存在明显重叠风险
* 是否会产生空动画或无效对象动画

### 3. 典型输出格式

```json
{
  "checker_version": "v1",
  "step_id": 3,
  "status": "needs_repair",
  "issues": [
    {
      "type": "missing_dependency",
      "action_index": 1,
      "message": "Point H is referenced before creation"
    }
  ],
  "repair_suggestions": [
    {
      "type": "insert_action_before",
      "target_action_index": 1,
      "new_action": {
        "action": "create_projection_point",
        "from": "B1",
        "to_line": "seg_BC",
        "new_point": "H"
      }
    }
  ]
}
```

### 4. 升级目标

让动画端从“边执行边猜”变成：

**先校验、再修补、再执行。**

## 八、新增模块三：可学习的数据闭环（Case Replay & Improvement Loop）

### 1. 为什么必须新增

当前项目如果不建立案例闭环，就会长期停留在“人工看视频发现问题，然后临时修一处”的状态。
这种方式不适合复杂题型迭代，也无法支撑模板库和策略引擎的持续优化。

### 2. 建议落盘的数据内容

每道题至少保存以下中间产物：

* 原题图
* OCR 结果
* 题型识别结果
* Geometry IR
* Teaching IR
* 命中的模板信息
* 辅助线候选及最终选择结果
* 动作可执行性校验结果
* 最终生成代码
* 渲染结果
* 执行日志
* 错误日志
* 人工修正记录

### 3. 数据闭环的价值

有了这一层，后续可以做：

* 哪类题型最容易失败
* 哪类动作最容易不可执行
* 哪种折叠模板命中率最高
* 哪种辅助线策略最有效
* 哪些步骤最常触发修补器
* 哪些失败案例值得沉淀成新模板

### 4. 建议新增案例记录结构

```json
{
  "case_id": "case_20260415_001",
  "problem_pattern": "fold_transform",
  "sub_pattern": "fold_with_perpendicular_construction",
  "geometry_ir_version": "v2",
  "teaching_ir_version": "v2",
  "template_match": {
    "template_id": "fold_with_perpendicular_construction",
    "confidence": 0.88
  },
  "execution_check": {
    "status": "repaired"
  },
  "render_result": {
    "status": "success"
  },
  "human_feedback": {
    "quality_score": 4,
    "issues": ["fold_path_not_natural"]
  }
}
```

### 5. 最终目标

形成真正可迭代的系统：

**失败样本不是丢失，而是变成模板、规则、校验器和策略引擎的训练数据。**

## 十、折叠题模板库（继续保留，并纳入题型识别层）

当前折叠题方向仍然是优先级最高的专项优化之一。
但在新方案中，它不再是单独漂浮的模块，而应归属于：

**题型识别层 → 模板族选择 → Teaching IR 生成**

建议折叠题模板族仍保留以下分类：

* `fold_point_to_point_distance`
* `fold_with_perpendicular_construction`
* `fold_then_similarity`
* `fold_then_area`

每个模板应明确：

* 适用题型与子题型
* 必选动作序列
* 必校验不变量
* 推荐辅助线候选
* 动作执行前置依赖
* 失败修补策略

---

## 十一、辅助线策略引擎升级（与新模块联动）

辅助线策略引擎继续保留三段式升级方向：

### 1. 候选生成

根据题型识别结果和 goal 生成辅助线候选。

### 2. 规则评分

综合考虑：

* 对目标的直接帮助程度
* 与现有关系的兼容性
* 图面复杂度成本
* 与模板动作序列的匹配度

### 3. 执行与验证

将候选动作写入 Teaching IR 后，再经过动作可执行性校验器检查，确认能够安全落地。

这意味着辅助线不再只是“策略建议”，而是主线动作的一部分。

## 十二、动画执行层补强点

在本轮新增模块之外，动画执行层仍建议继续补强以下能力：

* 面级翻折轨迹
* 辅助线出现 / 淡出时序标准化
* 直角标、等长标、对应角标注自动生成
* 标签避让
* 局部聚焦与关键区域放大

这些能力虽然不属于本轮新增三大模块，但会直接决定“像不像老师讲题”。

## 十三、建议的分阶段落地顺序

### 阶段A：题型识别层接入主线

目标：先让系统知道“这题属于哪种演示范式”。

产出：

* `ProblemPatternClassifier`
* 一级题型与子题型枚举
* 题型识别结果接入 `metadata`
* 模板匹配前置依赖改造

### 阶段B：动作可执行性校验器

目标：让 Teaching IR 从“看起来合理”升级为“可稳定执行”。

产出：

* `ActionExecutabilityChecker`
* 常见问题类型枚举
* 自动修补接口
* 执行前校验链路

### 阶段C：折叠题模板库升级

目标：折叠题不再依赖单点关键词命中，而是进入题型识别驱动的模板主线。

产出：

* 模板目录
* 模板匹配器
* 模板测试集
* 模板依赖约束声明

### 阶段D：辅助线策略引擎升级

目标：复杂题可主动构图，并让辅助线成为可解释、可执行、可验证动作。

产出：

* 候选生成器
* 打分器
* 执行写回机制
* 与动作校验器联动

### 阶段E：案例闭环系统

目标：形成案例回放、问题统计、模板迭代、规则优化基础设施。

产出：

* `case_record` 数据结构
* 中间产物自动落盘
* 失败样本聚类机制
* 人工修正回流机制

## 十四、本轮升级后的核心结论

在当前项目阶段，最重要的已经不是继续追加零散功能，而是建立真正的**结构化主线**。

本轮新增三项后，系统目标应正式调整为：

### 1. 题型先行

先判断“是什么题型、该怎么演”，再决定模板、动作和辅助线。

### 2. 动作先校验再执行

不再让动画层承担过多推断和纠错职责。

### 3. 每一次失败都可回流

让失败样本成为模板库、规则库和校验器的养料，而不是一次性损失。
